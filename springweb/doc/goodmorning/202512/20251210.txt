## II. 주관식 (기술형, 5문항)

**1. `@RequestParam`을 사용할 때 `defaultValue` 속성을 설정하지 않은 상태에서, 
	클라이언트가 해당 파라미터를 전송하지 않으면 어떤 문제가 발생하는지, 그리고 이를 해결하기 위해 `defaultValue`가 어떻게 동작하는지 서술하시오.**
	defaultValue 를 설정하지 않으면 해당 요청key로 요청하지 않으면 요청이 되지 않는다.
	defaultValue에 의해서 요청값이 없더라도 default값이 설정되기에 요청된다.

**2. 스프링에서 요청 파라미터를 자바 객체(DTO/VO)로 자동으로 매핑해주는 
	'커맨드 객체(Command Object)'가 동작하기 위해 지켜야 할 원칙인 'S.P.A'의 각 항목(S, P, A)이
	 무엇을 의미하는지 구체적으로 설명하시오.**
	 S(Setter) - DTO에 set메서드 선언
	 P(Parameter Name ) - 요청키와 set property의 이름과 타입이 맞아야 한다.
	 A(Automatic/Default Constructor) - DTO 객체에 기본 생성자를 반드시 선언하여야
	 									자동으로 매개변수로 객체가 생성되어 데이터를 전송한다.
	 									

**3. Controller 메서드에서 매개변수로 커맨드 객체(예: `Person p`)를 선언했을 때, 
	개발자가 명시적으로 `Model` 객체에 `addAttribute`를 하지 않아도 View에서 해당 데이터를 
	사용할 수 있는 이유(메커니즘)를 설명하시오.**
	
	스프링(MVC) 커맨드 객체로 요청 파라미터 바인딩을 성공적으로 완료하면, 해당 커맨드 객체를
	자동으로 Model객체에 추가한다. 이때 모델에 저장되는 기본 키(key) 값은 해당 클래스 이름의
	첫 글자를 소문자로 변환한 형태이다. Person 클래스의 객체는 person이란느 키로 모델에 추가된다.
	model.addAttribute() 호출 없이도 View(JSP)에서 el을 이용해 데이터에 접근할 수 있다.
	
	

**4. 단순 데이터 타입(String, int)을 받는 `@RequestParam` 방식과 객체(Object)를 
	사용하는 방식의 차이점을 '파라미터의 개수'와 '데이터 관리' 측면에서 비교하여 설명하시오.**
	파라미터 갯수 : 파라미터가 1개씩 늘어날 때마다 메서드 인자가 1개씩 늘어나 코드가 길어지고 복잡해진다.
	데이터 관리 : 데이터가 컨트롤러 메서드 내에서 개별 변수로 흩어져 관리되므로 재사용이나 그룹화가 어렵다.

**5. `Model` 인터페이스의 역할은 무엇이며, Controller와 View(JSP) 사이에서
